# MainEditorScreen 右侧信息栏重构计划

## 目标

将 `MainEditorScreen` 右侧信息栏重构为一个“正反双面”的统一信息板。

- 正面保持当前右栏的大体落位与视觉重量，不改变它作为主界面右侧核心信息区的职责。
- 正面只承担“战前快速决策信息”，背面承担“展开后的完整怪物说明”。
- 正反切换必须通过真实的翻面动画完成，不保留当前这种“悬赏图 + 独立 tooltip 面板”的分裂结构。
- 数据展示要直接来源于现有怪物定义、阶段推进与难度曲线乘算结果，不引入本地兜底文案、猜测性数值或临时后处理。

## 目标效果

### 正面

正面布局沿用当前右侧信息栏的大致空间分布，但内容调整为：

- 左上：阶段名
- 左下：怪物悬赏图片
- 右侧从上到下：
  - 怪物名字
  - 危险度
  - 赏金
  - `更多信息` 按钮

其中：

- 正面是“扫描式阅读”界面，强调一眼判断。
- `更多信息` 按下后，按钮进入变淡状态，信息板执行翻面，切换到背面。

### 背面

背面展示更完整的战斗信息：

- 怪物名字
- 怪物悬赏图片
- 由难度曲线乘算之后得到的基础属性
- 怪物技能描述

其中：

- “基础属性”必须是当前阶段真实参与战斗的最终基础属性展示，而不是怪物静态原始模板值。
- 技能描述沿用怪物定义中的正式描述来源，不额外拼接启发式说明。

## 当前实现基线

目前右栏相关实现分散在以下文件：

- `Scenes/main_editor_screen.tscn`
- `Scripts/UI/main_editor_screen.gd`
- `Scripts/Tests/ui_runner.gd`
- `Scripts/Tests/wanted_tooltip_style_runner.gd`

当前结构的核心问题：

- 右栏正面信息与“更多信息”实际上是拆成两块实现：
  - `NextMonsterPanel`
  - `MonsterTooltipPanel`
- 这会导致信息结构割裂，视觉上不像一个完整的信息板。
- 当前 tooltip 是悬浮展开逻辑，不是翻面逻辑，和这次目标不一致。
- `StageInfoPanel`、`RiskLabel`、`NodeLabel` 等历史残留信息与怪物信息板职责混杂，后续应重新归位。

## 重构原则

- 保持“一个信息板负责一类信息”的结构，不通过叠加遮罩、额外 tooltip、局部补丁来模拟双面设计。
- 正反两面共用同一块信息板容器，数据刷新与状态切换围绕单一组件组织。
- 翻面动画由明确的状态机驱动，不依赖 hover 触发或临时显隐拼接。
- 所有展示字段都从正式数据链路生成：
  - 当前阶段
  - 下一个怪物
  - 难度曲线乘算后的基础属性
  - 技能描述
- 不引入降级处理、fallback 文案、启发式估算或后处理补丁。

## 建议结构调整

### 场景层

建议把当前右栏怪物信息区重组为一个独立的双面信息板节点，例如：

```text
RightPanel
  MonsterInfoBoard
    BoardRoot
      FrontFace
        StageNameLabel
        WantedPosterRect
        MonsterNameLabel
        RiskLabel
        BountyLabel
        MoreInfoButton
      BackFace
        BackMonsterNameLabel
        BackWantedPosterRect
        FinalStatsLabel
        SkillDescriptionLabel
```

说明：

- `FrontFace` 和 `BackFace` 是同一信息板的两个面，不再让背面内容挂在独立 tooltip 容器里。
- 当前 `MonsterTooltipPanel` 应退出主结构。
- `StageInfoPanel` 是否保留，要根据阶段名是否完全并入正面后再决定；如果只剩重复信息，应移除而不是继续并存。

### 脚本层

建议在 `main_editor_screen.gd` 中把右栏怪物信息重构成一组明确职责的方法：

- `_refresh_monster_info_board()`
- `_refresh_monster_info_front()`
- `_refresh_monster_info_back()`
- `_set_monster_info_board_face(is_front: bool)`
- `_play_monster_info_flip(to_front: bool)`

同时收敛现有逻辑：

- `_refresh_next_monster_panel()` 的职责迁移到新的信息板刷新函数
- `wanted_poster_rect.mouse_entered / exited` 对 tooltip 的依赖移除
- `MonsterTooltipPanel` 相关状态控制删除

## 数据方案

### 正面数据

正面需要的数据字段：

- 阶段名
- 怪物名
- 危险度
- 赏金
- 悬赏图片

建议来源：

- 阶段名：当前关卡或下一战阶段显示逻辑的正式文本来源
- 怪物名：下一只怪物定义
- 危险度：沿用现有风险评估链路，不改判断算法，只改展示位置
- 赏金：沿用现有赏金展示链路
- 悬赏图片：沿用现有 wanted poster 素材映射

### 背面数据

背面需要的数据字段：

- 怪物名
- 悬赏图片
- 难度曲线乘算后的基础属性
- 技能描述

建议来源：

- 不直接显示怪物模板原始属性
- 通过当前阶段实际生成怪物数据时所走的同一套乘算逻辑，得到最终基础属性展示
- 技能描述直接取怪物正式定义文本

需要确认的一个实现重点：

- 如果当前代码里“下一战怪物预览”和“正式战斗怪物生成”不是同一数据构建入口，需要先抽出统一的怪物预览数据构造函数，保证正面摘要、背面详情、正式战斗三者看到的是同一只怪物的同一组数值语义。

## 动画方案

### 交互流程

1. 默认显示正面。
2. 点击 `更多信息`。
3. 按钮先进入变淡状态，避免仍像可继续点击的主按钮。
4. 信息板执行翻面动画。
5. 动画完成后显示背面。
6. 背面提供返回入口，翻回正面。

### 动画要求

- 动画必须是“翻面”而不是简单淡入淡出。
- 翻面期间只能有一个面处于主可见态，避免前后两面同时穿帮。
- 动画时长要短，目标是增强质感而不是拖慢读信息速度。
- 切面过程要锁定按钮重复点击，防止状态重入。

### Godot 实现建议

优先做法：

- 用单信息板容器驱动 `scale.x`、`pivot_offset`、正反面显隐切换的方式完成 2D 卡片翻面。
- 在缩至近似 0 的中点切换前后面显隐，再展开到目标面。

原因：

- 这符合当前 2D UI 项目结构。
- 不需要引入伪 3D 节点体系。
- 能稳定得到“正反翻转”的视觉结果，同时保持节点结构可维护。

避免的做法：

- 单纯 alpha 淡入淡出
- 背面面板从旁边滑入
- 用 tooltip 冒层伪装成翻页

这些都不符合这次“信息板翻面”的目标。

## 分阶段实施

### 第一阶段：结构收敛

- 在 `main_editor_screen.tscn` 中建立新的双面信息板结构
- 把当前正面怪物信息迁入 `FrontFace`
- 把 `MonsterTooltipPanel` 中的内容迁入 `BackFace`
- 明确 `StageInfoPanel` 的去留，避免阶段信息重复展示

交付标准：

- 不开启动画时，正反两面都能单独正确显示
- 当前 tooltip 依赖已被移除

### 第二阶段：数据统一

- 抽取“下一战怪物展示数据”构造逻辑
- 统一产出：
  - 名称
  - 悬赏图
  - 危险度
  - 赏金
  - 最终基础属性
  - 技能描述

交付标准：

- 正反两面显示的是同一只怪物的同一数据语义
- 背面属性已是难度曲线乘算后的结果

### 第三阶段：翻面交互

- 实现 `更多信息 -> 背面`
- 实现 `返回 -> 正面`
- 增加按钮变淡、动画中锁输入、动画后恢复状态

交付标准：

- 双向翻面稳定
- 不会因连点导致正反状态错乱

### 第四阶段：右栏整体收口

- 重新检查 `SynergyPanel`、`ActionButton`、剩余阶段信息在右栏中的排布
- 确保新信息板加入后，右栏视觉层级清晰
- 清理不再需要的旧节点与脚本引用

交付标准：

- 右栏没有重复信息区
- 没有历史节点残留占位

### 第五阶段：测试更新

需要同步改动的测试：

- `Scripts/Tests/ui_runner.gd`
- `Scripts/Tests/wanted_tooltip_style_runner.gd`

测试重点：

- 新信息板节点存在
- 默认显示正面
- 点击 `更多信息` 后最终显示背面
- 背面展示最终基础属性与技能描述
- 悬赏图仍根据怪物 id 正确切换
- 旧 `MonsterTooltipPanel` 路径断言移除或替换

## 风险与注意事项

- `ui_runner.gd` 当前仍对 `NodeLabel`、`RiskLabel` 的旧路径做断言，重构时必须同步调整。
- 现有文档记录显示右栏历史上发生过一次“计划删除但场景未完全收口”的情况，这次需要把结构和测试一起收干净。
- “最终基础属性”如果在现有代码中没有独立预览出口，最容易出现的错误是：
  - 正面摘要看的是预览数据
  - 背面看的是模板数据
  - 战斗实际又是第三套数据
  这必须通过统一构造函数避免。
- 翻面动画如果直接绑死在散落节点上，后续很难维护，建议从一开始就做成单信息板组件逻辑。

## 验收标准

- 右侧怪物信息区是一个统一的信息板，而不是“正面面板 + 独立 tooltip”。
- 正面展示：
  - 阶段名
  - 悬赏图片
  - 怪物名字
  - 危险度
  - 赏金
  - `更多信息` 按钮
- 点击 `更多信息` 后，按钮变淡，信息板执行翻面动画并展示背面。
- 背面展示：
  - 怪物名字
  - 悬赏图片
  - 难度曲线乘算后的基础属性
  - 怪物技能描述
- 前后两面数据一致、来源统一、无临时补丁逻辑。
- 相关自动化测试完成更新并通过。
