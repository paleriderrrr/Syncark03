# Acceptance Checklist

## PDF Compliance
- 角色数量必须是 `3`：战士、猎人、法师。
- 角色基础属性必须与 PDF 一致。
- 三个角色都必须有独立饭盒。
- 初始饭盒必须是 `3x3`。
- 饭盒最终范围必须限制在 `6x8`。
- 饭盒拓展必须按边连接，允许旋转、空洞、重排与改位置。
- 三角色共用食材仓库。
- 六大食物类别必须全部存在。
- 六类羁绊效果的基础值与成长公式必须全部实现。
- 每场战斗必须只有一个怪物。
- 战斗超过 `2` 分钟后双方持续掉血的规则必须实现。
- 默认承伤顺序必须是战士 -> 猎人 -> 法师。战斗中的站位是战士在前、猎人在中、法师最后。
- 固定关卡顺序必须与 PDF 一致。
- 初始金币必须是 `30`。
- 市场必须有 `5` 个栏位。
- 刷新价格曲线必须与 PDF 一致。
- 饭盒拓展刷出概率与售价必须与 PDF 一致。
- 食材购买数量范围与折扣规则必须与 PDF 一致。折扣曲线在合理范围内自由决定。
- 已写出的怪物数值与怪物技能必须与更新后的主策划一致。
- 食物系统的基础数值口径必须与用户最终指定的数值模型一致。
- 普通怪 `6` 只、Boss `1` 只的结构必须成立。
- 已有界面页面与 [08_ui_design_rules.md](/D:/2Projects/26.03.28%20Syncark03/08_ui_design_rules.md) 一致。
- 弹窗尺寸与基础结构必须符合 PDF 已知约束。

## Godot Compliance
- 项目组织必须符合 [09_godot_development_mode.md](/D:/2Projects/26.03.28%20Syncark03/09_godot_development_mode.md)。
- 主要页面必须可映射为 Godot 场景。
- UI 必须以 Godot 的 `Control` 体系为主。
- 数据驱动部分必须采用 Godot 友好的资源或数据文件组织方式。

## User-Confirmed Or Pending Items
- 危险度评级算法当前按“可合理自由设计”处理，若要进一步锁死公式必须再问用户。
- `54` 种食物的完整配置来源当前按 `食物清单.pdf` 处理，若开发时发现仍不足以逐条落地，必须再问用户。
- 香料类怪物未定稿内容当前允许留空，但若进入可战斗实现，缺失字段仍需再确认。
- UI 视觉细化项，如字体、主色、按钮风格、图标风格、动效规则，未明确前不得擅自决定。
- Godot 具体版本号未锁定，若进入项目初始化阶段需先询问用户。

## No-Assumption Check
- 实现过程中没有擅自删减 PDF 中已明确的系统。
- 实现过程中没有擅自新增 PDF 未要求的系统。
- 所有因资料不清而暂停的问题都已先向用户提问。
- 所有用户补充答复都已同步回文档和实现。
