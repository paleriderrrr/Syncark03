# Godot Development Mode

## Engine Override
- 当前开发引擎按用户最新要求切换为 `Godot`。
- 原 PDF 中的 Unity 版本号不再作为实现约束。
- 若需要锁定具体 Godot 版本，如 `4.2`、`4.3` 或 `4.4`，必须先询问用户。

## Godot-Oriented Project Shape
- 每个主要页面使用独立场景：
  - 标题页
  - 主编辑页
  - 设置页
  - 战斗页或战斗弹窗
  - 结算页
- 核心玩法逻辑与 UI 分离。
- 全局进度、关卡流转、仓库和当前冒险状态适合放在 `Autoload` 单例中。

## Node And UI Mode
- 界面主结构以 `Control` 节点体系实现。
- 弹窗优先使用 Godot 的窗口或弹窗组件实现。
- 饭盒编辑区可使用自定义网格控件或基于 `Control` 的拖拽网格。
- 三角色切换、市场刷新、战斗结算优先使用信号驱动。

## Data Mode
- 角色、怪物、食物、市场配置建议做成 Godot 易读的资源或结构化数据文件。
- `54` 种食物若要完整落地，必须保证数据结构可逐条配置：
  - 类别
  - 形状
  - 稀有度
  - 基础属性
  - 羁绊贡献
  - 价格
  - 掉落池归属
- 若数据来源仍不完整，必须先问用户，不得硬写占位表。

## Scene Responsibilities
- 主编辑页场景：
  - 三角色标签页
  - 饭盒编辑区
  - 仓库区
  - 金币与路线信息
  - 市场栏
  - 战斗入口
- 战斗场景或弹窗：
  - 单怪物展示
  - 三角色状态
  - 伤害数字
  - 回复与负面状态反馈
  - 结果结算

## Asset Import Mode
- 食物图块、怪物立绘、角色立绘和 UI 贴图按 Godot 资源导入管理。
- 需要保留 PNG 透明通道与像素边界清晰度。
- 若饭盒边框按 tile 组合实现，优先采用 Godot 易维护的切片方式组织资源。

## Implementation Restrictions
- 不允许因为 Godot 的实现习惯去修改 PDF 已确定的玩法结构。
- 不允许因为 Godot UI 组件限制而擅自删减界面要素。
- 不允许在未确认的情况下引入第三方 Godot 插件作为前置依赖。

## Must Ask The User Before Deciding
- Godot 具体版本号
- 是否允许使用插件
- 数据资源采用 `.tres/.res` 还是外部表格 / JSON 作为主来源
- 战斗页是独立场景还是主编辑页内弹窗
- 是否需要移动端兼容或只做 PC
