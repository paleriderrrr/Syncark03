# Scope And Priorities

## PDF-Mandated Scope
- 当前实现引擎按用户要求迁移为 `Godot`，但玩法、数值与界面约束仍按更新后的 PDF 执行。
- 三名角色：战士、猎人、法师。
- 三个独立饭盒，初始形状相同，均为 `3x3`。
- 饭盒可通过市场购买拓展块扩展，最终需限制在 `6x8` 空间内。
- 拓展块需要边连接，允许旋转、形成空洞、重新拼接、改变位置。
- 食材仓库三人共用，饭盒分别编辑。
- 六大食物类别：蔬果、甜品、肉类、饮品、主食、香料。
- 食物具有形状、特殊效果、触发条件、稀有度、属性加成、金币价值、类别。
- 同一角色饭盒中出现三种同类别食物时触发羁绊；效果层数按该类别在该饭盒中的总占格数成长。
- 自动战斗采用自走棋式实时演算。
- 每场战斗只有一个怪物。
- 默认承伤顺序为战士 -> 猎人 -> 法师。
- 战斗持续超过 `2 分钟` 后，双方开始等量持续掉血。
- 固定关卡顺序与市场/休整/战斗节奏必须按 PDF 实现。
- 普通怪共 6 只，最终 Boss 1 只。
- 市场系统必须包含食材出售、饭盒拓展出售、5 个商品栏位和刷新成本曲线。
- 战斗胜利后的金币与掉落包规则必须实现。
- 食物系统的基础等价模型与数值口径必须与更新后的 `食物清单.pdf` 对齐，或由用户明确指定采用数值改写版口径。

## Do Not Change Without User Approval
- 开发引擎的大版本选择
- 食物类别数量
- 角色数量与角色基础属性
- 关卡顺序
- 市场刷新公式
- 掉落包价值曲线
- 饭盒拓展概率与售价
- 普通怪与 Boss 数量
- 六类羁绊的基础值与成长公式

## PDF Conflicts Or Missing Details
以下内容不能自行决定，必须先问用户：

- 具体 Godot 版本号
- “战前系统与可计算性”仍然没有严格公式，除非沿用用户已确认的“自由决定”规则，否则需要再确认。
- `食物清单.pdf` 和数值改写版已给出统一数值口径，但若要落地完整 54 种食物逐条表，仍可能需要用户提供更完整的结构化源。
- 香料类普通怪的 `ATK`、攻击间隔与最终技能细节仍不完整。
- 美术页包含若干生产备注与 DDL，但不是所有尺寸和导出规则都具备实现级清晰度，落地前如有歧义需询问用户。

## Question-First Rule
- 只要出现“冲突”“待定”“未定义”“缺表”“描述不完整”五类情况，AI 必须暂停实施并提问用户。
- 在得到用户明确答复前，AI 不得用默认值、经验值或相似项目做替代。
